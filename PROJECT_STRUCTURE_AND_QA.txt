2. 🧠 LEARNING: WHY WE BUILT IT THIS WAY (Detailed Explanations)
------------------------------------------------------------------------

🟢 Why separate Backend and Frontend?
We use this decoupled architecture because it allows the Machine Learning (Python) to run freely in the background while the UI (JavaScript) stays blazing fast. If we combined them into one file, the heavy AI math would freeze the user interface every time it tried to calculate a prediction!

🟢 Why SQLite instead of MySQL/Postgres?
We use SQLite because it runs as a local file (`sql_app.db`). This makes the project highly portable and instantly runnable for academic/project reviews. Your examiners or professors can run the software without needing to install a heavy external database server.

🟢 Why generate "Synthetic" Data?
Real medical data regarding blood infections (sepsis) is protected by strict HIPAA privacy laws and is very hard to obtain. We use NumPy to synthetically generate 10,500 highly realistic mathematical ranges for heart rate, WBC, etc., mimicking real clinical scenarios to train the AI legally and safely!



3. ❓ PROJECT-BASED Q&A (Interview / Viva Preparation)
------------------------------------------------------------------------

Q1: Why do we use both Random Forest AND XGBoost?
A: Random Forest builds hundreds of independent decision trees simultaneously and votes on the result, which prevents overfitting. XGBoost builds trees sequentially, where each new tree specifically tries to fix the errors of the previous one. We use both so the system can contrast results; XGBoost mathematically performs better on skewed data, while Random Forest handles dense, standard data well.

Q2: What is "SMOTE", and why do we use it in ml_model.py?
A: In our dataset, we have 7,000 healthy patients and only 3,500 infected ones. This is called an "Imbalanced Dataset". If we train the AI natively, it might just learn to always guess "Healthy" to artificially get a high accuracy score! We use SMOTE (Synthetic Minority Over-sampling Technique) to mathematically clone and blend the infected cases so the AI learns both classes equally.

Q3: Why do we use "StandardScaler" before predicting?
A: Patient metrics have completely different ranges. WBC Count is around 7,000 to 15,000, while Hemoglobin is around 10 to 14. If we feed this directly to the AI, it might mathematically assume WBC is 1,000x more important just because the number is bigger! We use StandardScaler to squash all these numbers onto the exact same proportional scale centrally around 0.

Q4: How does the React app (frontend) talk to Python (backend)?
A: We use an asynchronous library called `axios` in `App.jsx`. When the user clicks the "PREDICT" button, `axios` bundles the patient's form data into a JSON package and fires an HTTP POST request over to `http://localhost:8000/api/predict`. FastAPI catches it, processes it, and sends the JSON result back!

Q5: Why did we implement a hard-coded "Safe Override"?
A: AI models don't have common sense; they just do math. If a patient comes in with perfectly healthy vitals across the board, there is a tiny statistical chance the model might randomly hallucinate an 'Infection' flag due to edge-case math. The Safe Override code acts as a safety net: "If WBC, Temp, BP, and Oxygen are all totally normal, force the result to Healthy", guaranteeing medical safety!
